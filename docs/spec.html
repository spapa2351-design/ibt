3. 시스템 구조

Newin이 단독으로 만드는 3개의 웹 프론트엔드 + 1개의 AI 엔진 + 1개의 결제 모듈.

📌 응시자용은 동일 웹 앱을 네이티브 쉘로 감싼다.
이유 5개
  1. 메인 사이트·관리자 백오피스·응시자 클라이언트는 사용자 주체와 권한이 다르므로 각각 별도의 프론트엔드.
  2. AI 엔진은 출제 보조·STT·채점·피드백 4가지 책임을 한 모듈에서 다룬다 — 일관된 평가 기준 학습이 핵심.
  3. 결제는 외부 일본 PG 1개를 연동하는 단일 모듈 — 메인 사이트 안에 흡수.
  4. 응시자 클라이언트는 단일 반응형 웹 앱. iOS/Android는 동일 코드를 WebView로 감싼 네이티브 쉘로 배포.
  5. 마이크/녹음은 Web API 1차 → 안정성 부족 시 네이티브 플러그인 fallback. 클라이언트 코드베이스 1개 유지.

6. 운영 정책 (Phase 1, Light)

1차 오픈 운영은 예외 상황을 자동화하지 않고 운영자가 수동 처리한다는 단순 원칙으로 통일.

① 환경 사전 점검

사전 점검을 통과한 데이터가 없으면 결제 버튼이 활성화되지 않는다.

점검: 마이크 권한·볼륨(데시벨)·5초 녹음 후 재생 — 3개 모두 통과 + 약관 동의 필수.

② 응시 진행

응시자는 시작·정지·중단 버튼을 가지지 않는다. 시간이 흐르면 자동으로 다음 단계로 이동.

지시문 자동 표시 → 준비 카운트다운(파랑) → 비프 → 녹음(빨강) → 자동 종료 → 자동 다음.

③ 자동 저장·재전송

네트워크가 끊겨도 응시는 계속되며, 답안은 단말 측 임시 저장 후 복구 시 자동 재전송.

저장 위치: 브라우저 IndexedDB / 앱 파일 시스템. 재전송 자동.

④ 시험 자동 종료

마지막 문항 녹음 시간 종료 → 자동 일괄 전송 → AI 채점 큐 등록.

수동 종료 버튼 없음. 응시 분리·일시정지는 라운드 2.

⑤ 결과·인증서 공개

AI 1차 채점은 시험 종료 즉시 진행되지만, 응시자 노출은 관리자가 "결과 공개" 클릭한 후.

평가자 2차 검수는 1차 오픈 미운영(GAP-06). 인증서 PDF는 위변조 검증 URL 포함.

⑥ 재응시 (Light)

시스템·네트워크 장애 한정. 운영자가 로그 확인 후 수동 티켓 발행.

단순 변심·결과 불만족은 1차 오픈 미허용.

⑦ 환불 (Light)

모두 운영 검토 수동. 자동 분기는 라운드 2.

접수 기간 내 자율 취소 / 시스템 장애 → 운영자 환불 처리. 응시 후 변심 미적용.

⑧ 회차·접수 (Light)

결제 후 14일 내 24시간 응시 가능 티켓 임시 운영. 회차 4회는 라운드 2 이후.

근거: 마스터 문서 — 트래픽 분산 + 클레임 분산 의도.

13.2 KoPASS에 시사하는 것
  1. 안내 사이트와 트랜잭션 사이트 분리 패턴이 일본 응시자에게 익숙. KoPASS도 단일 vs 분리 결정 필요.
  2. 현 TOPIK엔 앱이 없음. KoPASS의 "가입 → 마이페이지 → 앱 오픈" 동선은 차별화 포인트.
  3. 앱은 응시 전용 가능성. 회원·결제는 웹 단독으로 두면 PG·OS 정책 변경 영향 최소.
  4. 안내·접수 모두 일본어. KoPASS도 동일.
  5. 현 TOPIK은 성적·인증서를 한국 본사로 빠지지만, KoPASS는 일본 단독. 마이페이지가 더 무거워짐.
