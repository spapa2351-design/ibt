# 유저 플로우 + 서비스 웹사이트 구조 (1차 오픈, Light)

> 본 문서는 1차 오픈 범위에 한정한 가벼운 골격이다. 환불 분기·재응시 티켓·딥링크 같은 무거운 동선은 *1차 오픈 범위 외* 섹션에서 목록만 둔다.

## 한 줄

**1차 오픈은 *가입 → 접수 → 결제 → 응시 → 결과·인증서* 한 동선으로 충분하다.**

## 이 한 줄을 지지하는 이유 5개

1. 최근 합의 문서(`Kopass 서비스 개발 기능 명세.pdf`)가 메인·관리자·응시자 3축을 단순한 4섹션으로 정리해 두었으므로 그걸 따라간다.
2. 관리자 측은 기존 솔루션을 활용하므로 **대메뉴 식별만으로 1차 오픈에 충분**하고, 추가 디테일은 문제 만들기 시점에 깊어진다.
3. 응시 콘텐츠(말하기 1~6형)는 본 라운드에서 의도적으로 보류하므로, 응시자 플로우도 *환경 점검 → 응시 → 결과*의 골격만 잡는다.
4. 보조 동선(환불·재응시·탈퇴·1:1 문의)은 정책 결정과 외부 협상이 끼어 있으므로, 이름만 박고 정의는 라운드 2 이후로 미룬다.
5. 라이트하게 잡힌 골격은 디자인·개발 발주를 빠르게 시작할 수 있게 해주고, 추가 디테일은 골격 위에 *덧붙이는* 방식으로 자라야 일관성이 유지된다.

## 2. 서비스 웹사이트 — 사이트맵 (PDF 4블록)

```
KoPASS 서비스 웹사이트
│
├─ 회원관리
│   ├─ 회원가입 (소셜 로그인 옵션 검토)
│   ├─ 로그인
│   └─ 계정 찾기 / 비밀번호 재설정
│
├─ 상품·결제
│   ├─ 시험 목록 (검색·필터)
│   ├─ 시험 상세
│   └─ 결제 (외부 일본 PG 1개 연동)
│
├─ 마이페이지
│   ├─ 구매한 시험 목록 / 응시 진입
│   ├─ 결제 내역
│   ├─ 환불 신청
│   ├─ 회원정보 수정 / 탈퇴
│   └─ 인증서 다운로드
│
└─ 고객센터
    ├─ 공지 / FAQ
    └─ 1:1 문의
```

> URL 정의·메뉴 라벨 일본어 표기·페이지 분리 단위는 라운드 2 `06-screen-inventory`에서.

## 3. 관리자 — 대메뉴만

PDF의 5개 대메뉴 그대로 식별:

1. **시험 관리** — 시험 등록·공개, 응시 조건, 시험지 구성
2. **평가 및 채점** — 응시자별 채점, AI STT 결과 확인·수정, 점수 보정·확정
3. **회원 및 권한** — 회원 조회, 관리자 등급 권한
4. **통계 및 분석** — 응시 통계, AI 평가 품질
5. **시스템 설정·보안** — 인증서 템플릿, 약관, 2FA, 활동 로그

> **운영 메모**:
> - 관리자 측은 **기존 솔루션을 활용**하므로 1차 오픈은 대메뉴 식별만으로 충분.
> - 향후 핵심은 **문제 만들기** (LLM 자동 생성 보조·에디터 수정·문제 은행). 본 영역은 별도 라운드에서 깊게 다룬다.

## 4. 응시자 — 4섹션 (PDF 그대로, 한 줄씩)

| 섹션 | 핵심 |
|---|---|
| **시험 준비** | 진행 중 시험·이력 조회, 시험 안내 정보, 환경 점검(마이크·네트워크·배경소음) |
| **시험 응시** | 문항별 타이머·자동 녹음, 답안 제출·서버 전송 |
| **결과 확인** | 채점 완료 알림, 항목별 상세 리포트·피드백, 재응시·모범 답안 |
| **인증서 발급** | 성적 기반 인증서, 위변조 진위 확인, PDF 다운로드 |

> 반응형 뷰포트(데스크탑 2단 / 모바일 1단)·말하기 문제 유형별 인터랙션은 라운드 2.

## 5. 1차 오픈 핵심 플로우 (2개)

### 5.1 가입 → 응시

```
방문자 → 시험 목록 → 시험 상세 → [회원가입·로그인] → 환경 사전 점검(통과 필수)
       → 결제(PG) → 마이페이지 반영 → "응시하기" → 응시 진행 → 자동 제출
```

### 5.2 결과 → 인증서

```
AI 1차 채점 → 채점 완료 알림(이메일·마이페이지) → 결과 확인(점수·등급·5각 차트·문항별 피드백)
            → 인증서 PDF 다운로드
```

> 결제 실패·환경 점검 미통과·네트워크 끊김·재응시 같은 *예외 분기*는 라운드 2 정책 정의와 함께.
