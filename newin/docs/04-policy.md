# 운영 정책 (Phase 1, Light)

> 본 문서는 **1차 오픈 시점의 운영 룰**만 담는다. 자동 분기·매트릭스·정책 매크로 같은 무거운 운영 도구는 라운드 2 이후.

## 한 줄

**1차 오픈 운영은 *예외 상황을 자동화하지 않고 운영자가 수동으로 처리한다*는 단순 원칙으로 통일한다.**

## 이 한 줄을 지지하는 이유 5개

1. 1차 오픈은 응시자 데이터·장애 패턴이 아직 쌓이지 않아 자동 분기를 설계할 통계가 없다.
2. 일본인 응시자는 안정성·예측 가능성을 중시하므로 *시스템이 알아서 결정*하는 분기보다 *운영자가 보고 결정*하는 흐름이 신뢰가 높다.
3. 자동 분기는 한 번 잘못 박히면 환불·재응시 분쟁의 자동 책임 인정이 되어버린다 — 1차 오픈에서 가장 피해야 하는 리스크.
4. 운영자가 직접 처리하는 케이스가 곧 라운드 2의 자동화 룰을 만드는 데이터다 — 수동 처리 로그가 다음 라운드의 정책 근거가 된다.
5. 라이트한 정책은 디자인·개발 발주를 가볍게 해서 9월 런칭 일정 위험을 줄인다.

---

## 정책 1 — 환경 사전 점검

**핵심 룰**: 사전 점검을 통과한 데이터가 없으면 결제 버튼이 활성화되지 않는다.

| 항목 | 1차 오픈 적용 |
|---|---|
| 점검 항목 | 마이크 권한 / 볼륨(데시벨 미터 시각화) / 5초 녹음 후 즉시 재생 / 네트워크 |
| 통과 조건 | 4개 항목 모두 통과 + "통과 후 발생한 개인 기기 문제는 사용자 책임" 약관 동의 |
| 미통과 처리 | 결제 버튼 비활성화. 응시자에게 *어떤 항목에서 막혔는지* 명시 |
| 사전 점검과 본 시험 시간 갭 | **GAP-08**로 트래킹. 1차 오픈은 응시 시작 직전 *재테스트 미강제* (가정), 운영 데이터 본 후 라운드 2에서 결정 |

**라운드 2 이후**: 디바이스 핑거프린팅 기반 재테스트 자동화, 마이크 변경 감지.

**근거**: 마스터 문서 *"결제 전 필수 환경 테스트 강제 수행, 테스트 통과 데이터 없으면 결제 버튼 비활성화"*

---

## 정책 2 — 응시 진행

**핵심 룰**: 응시자는 *시작·정지·중단 버튼을 가지지 않는다*. 시간이 흐르면 자동으로 다음 단계로 이동한다.

| 단계 | 1차 오픈 적용 |
|---|---|
| 지시문 | 자동 표시 + 음성(TTS 또는 mp3) 자동 재생 |
| 준비 시간 | 파란 타이머 자동 카운트다운 + 메모장 활성화 |
| 비프음 | 준비 시간 종료 시 자동 발생 |
| 녹음 시간 | 빨간 타이머 자동 카운트다운 + "녹음 중" 인디케이터 + Waveform |
| 녹음 종료 | 시간 종료 시 자동 종료 + 자동 저장 + 다음 문항으로 자동 이동 |
| 응시자 조작 | *없음*. "이전 문항으로 돌아가기"·"멈추기"·"건너뛰기" 버튼 없음 |

**라운드 2 이후**: 응시 중 일시정지(전화 인터럽션 등) 정책. 1차 오픈은 인터럽션 발생 시 *시스템 장애 재응시 티켓* 흐름으로 흡수.

**근거**: 마스터 문서 *"별도 조작을 하지 않아도 시간 종료 시 자동 다음 문제로 이동"*. 실증 페인포인트(녹음 타이밍 불명확 28/65)도 *자동화 강화*가 답.

---

## 정책 3 — 자동 저장·재전송

**핵심 룰**: 네트워크가 끊겨도 응시는 계속 진행되며, 답안은 단말 측 임시 저장 후 복구 시 자동 재전송된다.

| 항목 | 1차 오픈 적용 |
|---|---|
| 저장 시점 | 문항별 녹음 종료 직후 (자동) |
| 저장 위치 | 단말 측 로컬 (브라우저 IndexedDB / 앱 파일 시스템) |
| 재전송 | 네트워크 복구 감지 시 자동. 응시자 조작 없음 |
| 시험 종료 시 미전송 답안 | 자동 일괄 전송 시도. 끝까지 실패 시 → 시스템 장애 재응시 티켓 흐름 |
| 응시자 알림 | *오프라인 표시는 하지만 응시 중단을 강요하지 않는다* |

**라운드 2 이후**: 백그라운드 동기화 정책 강화, 부분 전송 후 단말 변경 시나리오.

**근거**: 마스터 문서 *"임시 저장 기능 도입 검토 (네트워크 끊김 등의 대비)"*. 실증 데이터 — *"문제를 보거나 답변을 준비하는 동안 LINE 화면이 닫혀 시간 손실"*은 MaroPass에서는 자동 저장으로 해결.

---

## 정책 4 — 시험 자동 종료

**핵심 룰**: 응시자가 마지막 문항의 녹음 시간을 다 쓰면 *자동으로 시험이 종료되고 모든 답안이 일괄 전송*된다.

| 항목 | 1차 오픈 적용 |
|---|---|
| 종료 트리거 | 마지막 문항 녹음 시간 종료 |
| 종료 동작 | 자동 일괄 전송 → 응시 종료 화면 → AI 채점 큐 등록 |
| 수동 종료 | 없음. 응시자가 중도 종료할 수 있는 버튼 없음 |
| 중도 이탈 | *마이페이지에서 정의된 응시 윈도우* 안에 재접속해야 이어서 응시 가능 (Q14 — 1차 오픈은 이어서 응시 미허용 가능, 정책 결정 필요) |

**라운드 2 이후**: 응시 시간 일시정지 / 이어서 응시 / 응시 분리(여러 세션) 정책.

**근거**: 마스터 문서 *"응시자가 '전송' 버튼을 누르지 않아도 시간 종료 시 자동 저장 및 전송"*

---

## 정책 5 — 결과·인증서 공개

**핵심 룰**: AI 1차 채점은 시험 종료 즉시 진행되지만, 응시자 노출은 *관리자가 결과 공개 버튼을 누른 후*에 이뤄진다.

| 항목 | 1차 오픈 적용 |
|---|---|
| AI 1차 채점 | 시험 종료 즉시 자동 |
| 응시자 노출 | 관리자가 *백오피스에서 "결과 공개" 클릭* 후 마이페이지에 노출 + 이메일 알림 |
| 평가자 2차 검수 | 1차 오픈에서는 운영자가 수동 검수 (Premium 미운영). 2차 채점 운영 시점은 GAP-06 |
| 인증서 PDF | 결과 공개 후 마이페이지에서 다운로드. 위변조 검증 URL 포함 |
| 답안 초기화 | 관리자만 가능. 답안 초기화 시 응시자는 재응시 가능 |

**라운드 2 이후**: 결과 공개 자동화(채점 후 N분), 평가자 UI 도입, 인증서 템플릿 다중화.

**근거**: 마스터 문서 *"성적 발표 예정일에 성적 조회 버튼 활성화"*. 디자인 — 응시자 관리 백오피스의 "결과 공개" 버튼.

---

## 정책 6 — 재응시 (Light)

**핵심 룰**: 1차 오픈에서 재응시는 *시스템·네트워크 장애 케이스 한정*. 운영자가 로그를 확인하고 수동으로 재응시 티켓을 발행한다.

| 케이스 | 1차 오픈 적용 |
|---|---|
| 시스템 장애로 응시 불가 | 운영자 로그 확인 → 재응시 티켓 발행 → 응시자 마이페이지에 새 응시 인스턴스 등록 |
| 네트워크 끊김으로 일부 답안 미전송 | 자동 재전송 → 실패 시 시스템 장애 케이스로 흡수 |
| 응시자 단순 변심 | **재응시 미허용** (1차 오픈) |
| 동일 시험 결과 불만족 | **재응시 미허용** (1차 오픈) |

**라운드 2 이후**: 응시자 자율 재응시 정책 (횟수 제한, 추가 결제 등), 자동 발행 룰.

**근거**: 마스터 문서 *"시스템 또는 네트워크 에러로 평가가 정상적으로 진행되지 못한 경우 개발사에서 로그 확인 후 재응시 티켓을 발행"*

---

## 정책 7 — 환불 (Light)

**핵심 룰**: 1차 오픈 환불은 *모두 운영 검토 수동 처리*. 자동 분기는 라운드 2 이후.

| 케이스 | 1차 오픈 적용 |
|---|---|
| 접수 기간 내 자율 취소 | 응시자가 마이페이지에서 신청 → 운영 검토 → 승인 시 PG 환불 호출 |
| 시스템 장애 (재응시도 불가한 경우) | 운영자가 직접 환불 처리 (응시자 신청 불요) |
| 응시 후 변심 | **환불 미적용** |
| 결제 실패·이중 결제 | 결제 모듈에서 자동 회수 (PG 트랜잭션 정합성) |

**라운드 2 이후**: 자동 환불 분기 (접수 기간 내 + 미응시 → 자동 승인), 환불 사유 코드 매트릭스.

**근거**: 마스터 문서 *"접수 기간 내 취소하거나 시스템 장애 등 주관사의 귀책사유 발생 시 전액 환불"*. *(일본 내 TOEFL/TOEIC 정책 준용)*

---

## 정책 8 — 회차·접수 (Light)

**핵심 룰**: 1차 오픈은 **결제 후 14일 내 24시간 응시 가능 티켓** 임시 운영 방식.

| 항목 | 1차 오픈 적용 |
|---|---|
| 접수 방식 | 상시 접수 + 결제 즉시 응시 가능 |
| 응시 윈도우 | 결제일로부터 14일 |
| 응시 윈도우 내 응시 횟수 | 1회 (재응시는 정책 6 케이스 한정) |
| 회차 윈도우 운영 (3·6·9·12월 2주씩) | **1차 오픈 미운영**. 라운드 2 이후 검토 |

**라운드 2 이후**: 문제 은행·평가 체계 안정화 후 *상시 접수* 또는 *회차 윈도우 운영*으로 전환.

**근거**: 마스터 문서 *"결제 후 14일 이내 24시간 언제든 응시 가능한 티켓 판매 방식으로 전환하여 트래픽 분산. 문제 은행 및 평가 장치가 안정화될 때까지 임시적으로 시행"*

## 정책 9 — 회원 자격 요구

**한 줄 룰**: 무료 샘플 응시·본 시험 결제는 **로그인 회원만** 가능하다. 비로그인 사용자는 클릭 시 회원가입 페이지로 이동하며, 가입 완료 후 원래 동작으로 자동 복귀한다.

**이유 5가지**
1. **어뷰징 방지** — 비로그인 무제한 샘플은 서버 비용과 채점 데이터 오염을 초래
2. **리드 확보** — 무료 체험은 가장 강력한 유입 자원. 가입 없이 가져가게 두면 마케팅 비용이 누수됨
3. **결과 보관** — 샘플·본 시험 결과를 마이페이지에 남기려면 user 식별 필수. 비회원에겐 결과 일회용
4. **유료 전환 트리거** — "샘플 결과 → 본 시험 결제 유도" CTA는 회원에게만 발화 가능
5. **CS 추적·약관 동의 시점** — 응시 중 사고 클레임 대응에 user 식별 필요. 수집·이용 동의는 회원가입 시점에 받는 게 깔끔

**구현 규칙**
- "무료 샘플 응시" / "바로 구매하기" 클릭 시 로그인 상태 확인
- 비로그인 → `ui-auth.html?intent={sample|purchase}&return={원래 진행할 URL 또는 마커}` 로 이동
- 회원가입 페이지 상단에 intent 배지 표시 ("🎁 무료 체험을 시작하려면 회원가입을 완료해 주세요" / "💳 결제를 위해 회원가입을 완료해 주세요")
- 가입 완료 후 intent에 따라 원래 동작 자동 진행 (intent=sample → 시험 상세로 복귀 후 환경 점검 자동 발화 / intent=purchase → 결제 페이지로 이동)
- 본 정책은 **UI 게이트** — 실제 보안은 API 레벨에서 인증 토큰 검사로 별도 보장 (`06-feature-spec.md` 참조)

**예외**
- 시험 상세 페이지 열람 자체는 비로그인 허용 (SEO·홍보 목적)
- 샘플 듣기 (음성 미리듣기) 등 가벼운 미리보기 액션은 비로그인 허용

## 정책 10 — MaroPass portal + 터치클래스 자산 재사용 (새 창은 UI 수단)

**한 줄 룰**: MaroPass는 **결제·환불·인증서 portal**이고, 응시·결과·시험 관리는 **기존 터치클래스 사용자 web·관리자 web** 위에 MaroPass 특화 영역만 신설한다. 학습자에게는 portal과 사용자 web을 **새 창**으로 분리해 노출한다.

**이유 5가지**
1. **기존 자산 재사용** — 응시 클라이언트·인증·시험 관리·문제 은행 같은 기 검증된 터치클래스 인프라를 그대로 활용해 개발 표면적·일정 위험을 1/3로 축소. 신규 개발은 portal과 MaroPass 특화 영역(말하기 응시·5영역 문제 생성·MaroPass 관리 메뉴)에만 집중.
2. **브랜드 자산 vs 학습 자산 분리** — 결제 영수증(특정상거래법·적격청구서)·인증서(위변조 검증 URL이 `kopass.*` 도메인이어야 신뢰 성립)는 MaroPass portal에. 응시 데이터·결과는 터치클래스에 누적되어 학습 이력과 같은 공간에 쌓임.
3. **앱 확장 전제** — 사용자 web을 분리해두면, 라운드 2 이후 현재 운영 중인 터치클래스 app 안에서 MaroPass 응시 클라이언트도 동일 코드 베이스로 wrapping 가능. portal은 결제·인증서 영역이라 웹 단독으로 충분.
4. **운영 콘솔 일원화** — 운영자는 기존 터치클래스 관리자 web에서 시험 관리·문제 은행·응시자 관리를 그대로 사용. MaroPass portal에서 발생한 가입·결제·환불·인증서 데이터는 신규 MaroPass 메뉴 5개로만 추가되어 단일 콘솔에서 처리.
5. **미래 학습 동선 연결** — 결과 화면이 터치클래스 사용자 web 안에 있어 응시 → 5각 약점 진단 → 추천 학습 콘텐츠로 자연 연결. 라운드 2 이후 터치클래스 app 내 한국어 코스와 연동해 응시→학습 동선 완결.

**구현 규칙**
- MaroPass portal 마이페이지 "응시하기" → `window.open()`으로 사용자 web home (새 창)
- 새 창 진입 시 SSO 토큰으로 사용자 web에 자동 로그인 (portal 로그인 세션 공유)
- 안내 문구: **"응시는 새 창에서 진행됩니다. 팝업이 차단되면 허용해 주세요."**
- 팝업 차단 감지 시 사용자에게 재시도 가이드 표시
- 사용자 web 결과 확인 화면 하단에 "인증서·결과지 PDF 다운로드는 MaroPass portal 마이페이지에서" 안내 + 이동 링크
