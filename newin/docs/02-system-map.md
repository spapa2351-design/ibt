# 시스템 맵

## 한 줄

**MaroPass는 결제·환불·인증서 portal(신규)이고, 응시·결과·시험 관리는 기존 터치클래스 자산 위에 MaroPass 특화 영역만 신설한다.**

## 이 한 줄을 지지하는 이유 5개

1. **기존 자산 재사용**으로 개발 표면적·일정 위험을 1/3로 축소. 응시 클라이언트·인증·시험 관리·문제 은행 같은 기 검증된 터치클래스 인프라를 그대로 활용하고, 신규 개발은 MaroPass portal과 특화 영역(말하기 응시·5영역 문제 생성·MaroPass 메뉴)에만 집중.
2. **브랜드 자산 vs 학습 자산 분리** — 결제 영수증(특정상거래법·적격청구서)·인증서(위변조 검증 URL이 `kopass.*` 도메인이어야 신뢰 성립)는 MaroPass portal에. 응시 데이터·결과는 터치클래스에 누적되어 학습 이력과 같은 공간에 쌓임.
3. **앱 확장 전제** — 사용자 web을 분리해두면, 라운드 2 이후 현재 운영 중인 터치클래스 app 안에서 MaroPass 응시 클라이언트도 동일 코드 베이스로 wrapping 가능. portal은 결제·인증서 영역이라 굳이 app에 들어갈 필요 없이 웹 단독으로 충분.
4. **운영 콘솔 일원화** — 운영자는 기존 터치클래스 관리자 web에서 시험 관리·문제 은행·응시자 관리를 그대로 사용. MaroPass portal에서 발생한 가입·결제·환불·인증서 데이터는 신규 MaroPass 메뉴 5개(회원·결제·환불 승인·결과 공개·인증서 템플릿)로만 추가되어 단일 콘솔에서 처리.
5. **미래 학습 동선 연결** — 결과 화면이 터치클래스 사용자 web 안에 있으므로 응시 → 5각 약점 진단 → 추천 학습 콘텐츠로 자연 연결. 라운드 2 이후 터치클래스 app 내 한국어 코스와 연동해 응시→학습 동선 완결.

**핵심 원칙**
- **신규 = MaroPass portal 한 개** (결제·환불·인증서·1:1 문의·FAQ)
- **기존 재사용 = 사용자 web · 관리자 web** — 각각 MaroPass 시험 응시 영역, MaroPass 메뉴 5개만 추가
- **새 창은 UI 수단**이지 본질이 아님. 본질은 **소유권·책임 분리** (portal = 보관 자료 / 사용자 web = 학습 이력)
- 회원·결제 데이터 흐름 = portal 가입·결제 → 관리자 web 신규 "MaroPass 회원" 메뉴로 집계

> 외부 콘텐츠(출제 기준·예시 문항·평가 기준·운영 정책)는 외부에서 전달받는 입력으로 전제.

## 디바이스별 진입 방식 (1차 오픈)

| 디바이스 | 진입 방식 | 뷰포트 | 비고 |
|---|---|---|---|
| 데스크탑 (Win/Mac) | 브라우저 (Chrome/Safari/Edge) | **데스크탑 뷰포트** (2단 — 좌: 문제, 우: 응시) | 사용자 주력. 마이크 권한·녹음 안정성 가장 높음 |
| 모바일 웹 | 모바일 Safari/Chrome | **모바일 뷰포트** (1단 스택) | 폴백 + 모바일 사용자 |
| iOS·Android 앱 | App Store / Play Store | — | **라운드 2 이후 도입**. 푸시 알림 인프라도 함께 |

### 마이크·녹음 fallback 전략

1. **1차**: Web API (`MediaRecorder`, `getUserMedia`) — 데스크탑·모바일 웹 동일 코드
2. **2차 (라운드 2 앱 도입 시)**: 네이티브 쉘 안에서 네이티브 마이크 플러그인으로 우회
3. 검증 포인트: 모바일 Safari 백그라운드 녹음 안정성, 권한 요청 UX, 인터럽션 처리

## 시스템별 책임 경계

### 1. 서비스 사이트 (MaroPass portal — ★ 신규)

> **결제·환불·인증서 portal**. 응시 자체는 담당하지 않고, 보관 자료(영수증·인증서·결과지 PDF) 다운로드 + 회원·환불·1:1 문의의 단일 접점.



| 영역 | 포함 |
|---|---|
| 회원 관리 | 회원가입(이메일·LINE·Google·Apple), 로그인, 비밀번호 재설정 |
| 시험 접수 | 시험 목록·필터, 시험 상세, 무료 샘플 응시 진입, 결제 |
| 마이페이지 | **응시하기 버튼**(새창으로 사용자 web), 응시할 시험, 결제 내역(영수증 PDF), **인증서·결과지 PDF 다운로드**, 결과 보기 진입, 환불 요청, 회원정보 수정·탈퇴 |
| 고객센터 | 공지·FAQ, 1:1 문의, 약관·개인정보·특정상거래법 표시 |
| 회원 게이트 | 무료 샘플 응시·바로 구매하기 클릭 시 로그인 상태 검사 → 비로그인 시 가입 페이지로 (정책 9) |

### 2. 터치클래스 사용자 web (◇ 기존 + MaroPass 응시 영역 추가)

> **기존 운영 중인 터치클래스 사용자 web을 그대로 활용**. 신규는 MaroPass 결제분 시험만 노출되는 home과 말하기 IBT 5영역 응시 클라이언트뿐. 본인 확인·환경 점검 게이트·자동 저장/재전송 같은 인프라는 기존 그대로.



| 영역 | 포함 |
|---|---|
| home | 결제된 시험 목록(응시 대기 + 응시 완료). 서비스 사이트 마이페이지 "응시하기" 클릭 시 진입 |
| 본인 확인 | 이름·이메일·시험명 표시 + 본인 확인 체크박스 (카메라·SMS는 라운드 2) |
| 환경 점검 게이트 | 마이크·볼륨·5초 리플레이 3단계 점검 (1차 오픈. v1 보고용 4개 항목에서 운영 결정으로 축소) |
| 응시 클라이언트 | 데스크탑 2단(좌:문제, 우:응시) / 모바일 1단. 타이머, 자동 녹음, 메모장, 자동 저장·전송 |
| 응시 중 사고 대응 | 네트워크 끊김 자동 재연결+시간 보존, 마이크 권한 회수 가드, 타이머 자동 제출, 이탈 차단 경고 |
| 결과 확인 | 운영자 결과 공개 후 점수표·1급~6급 등급·문항별 AI 피드백 노출 |
| PDF 다운로드 안내 | "인증서·결과지 PDF는 서비스 사이트 마이페이지에서 다운로드"로 이동 안내 |

### 3. 터치클래스 관리자 web (◇ 기존 + MaroPass 메뉴 5개 추가)

> **시험 관리·문제 은행·응시자 관리·평가/채점·통계·시스템은 기존 그대로**. MaroPass 신규는 portal에서 흘러들어오는 회원·결제·환불·결과 공개·인증서 템플릿 메뉴 5개와, 문제 만들기 내 MaroPass 말하기 5영역 탭 추가뿐.



| 영역 | 포함 |
|---|---|
| ★ 회원 목록 (신규) | 서비스 사이트에서 발생한 가입·결제·환불 데이터 집계. 단일 화면에서 회원 정보·결제 이력·응시 이력 확인 |
| 시험 관리 | 시험 개요 등록, 공개 상태, 응시 조건/옵션, 응시 기간 |
| 문제 만들기 | 5영역(짧은 문장 읽기·질문 대답·생활 속 대화·그림 설명·화용 과제) + 등급·10개 카테고리 칩 |
| 문제 은행 | 분류 트리(초급/중상급·1형/2형) + 주제 다중 칩 필터 + 주제 컬럼 표시 |
| 시험지 관리 | N문항 단위 시험지 세트 구성, Drag & Drop 정렬, 배점 일괄 설정 |
| 응시자 관리 | 시험별 응시자 목록·결과·답안 초기화·결과 공개·CSV 다운로드 |
| 평가/채점 | AI 1차 자동 채점 + 운영자 보정. 평가자 UI(2차 채점)는 라운드 2 |
| 결제 관리 | 결제일·응시자·금액·상태(결제완료·환불요청·환불완료·취소·실패) + 영수증 + 환불 처리 |
| 통계 | 응시 통계, 문항별 정답률, AI 평가 품질 |
| 시스템 | 인증서 템플릿, 약관/개인정보처리방침, 2FA(라운드 3), 활동 로그 |

### 4. AI 엔진

| 영역 | 포함 |
|---|---|
| 출제 보조 | 유형·주제·난이도·필수 문법 입력 → LLM이 지문·TTS 스크립트·모범 답안·채점 기준표 자동 생성 (라운드 2 핵심) |
| STT | 응시자 녹음 → 텍스트 변환 |
| 채점 | 발음·유창성·어휘·문법·내용 구성 점수 산출 + 1급~6급 등급 매핑 |
| 피드백 생성 | 문항별 STT 비교 + 추천 연습 (자음/받침/억양) |

### 5. 결제 모듈

| 영역 | 포함 |
|---|---|
| 외부 PG 연동 | 일본 PG 1개사 연동 예정 (사명 미확정 — GAP-12) |
| 결제 수단 | 신용카드 + QR 결제 |
| 영수증 | PDF 발행 (서비스 사이트 마이페이지) |
| 인보이스 | 적격청구서 PDF 발행은 PG API 연동 시점 → 라운드 2 |
| 환불 처리 | 운영 정책에 따른 환불 트랜잭션 |

## 점선 영역 (미결정)

1. **일본 PG 1개사 선정**: GAP-12.
2. **마이크/녹음 — 모바일 웹 안정성 검증**: 라운드 2 앱 도입 시점에 네이티브 fallback 기준 재정의.
3. **평가자 UI 런칭 시점**: 2차 채점 운영 시점에 종속. 라운드 2.
4. **새 창 팝업 차단 대응 UX**: 1차 오픈 직전 운영 시뮬레이션 필요.
