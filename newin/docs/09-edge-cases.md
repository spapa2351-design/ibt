# 엣지케이스 (Phase 1, Light)

> 본 문서는 **1차 오픈 시점에 시스템·운영이 마주칠 경계 케이스**만 담는다. 본문 정책(`04-policy.md`)이 *정상 흐름* 룰이라면, 본 문서는 *정상 흐름이 깨지는 지점*에 대한 답이다. 자동 분기·매트릭스·룰 엔진은 라운드 2 이후.

## 한 줄

**1차 오픈 엣지케이스는 *시스템이 흡수할 수 있는 케이스는 자동 흡수하고, 나머지는 운영 SLA로 받는다*는 단순 원칙으로 통일한다.**

## 이 한 줄을 지지하는 이유 5개

1. 1차 오픈은 응시 데이터·장애 패턴이 쌓이지 않아 *자동 분기 룰*을 설계할 통계가 없다 — `04-policy.md` 정책 1차 오픈 한 줄과 같은 근거.
2. 시스템이 *흡수*하는 케이스(자동 저장·자동 재전송·자동 종료)와 운영이 *수동 처리*하는 케이스(재응시 티켓·환불·결과 공개)를 명확히 가르면, 응시자가 마주치는 분기가 사라진다 — 일본인 응시자는 *예측 가능성*을 중시한다.
3. 운영 SLA로 받는 케이스에는 *영업일 N일* 같은 응답 시간만 명시하면 운영팀 캐파 한도 안에서 1차 오픈 운영이 가능하다.
4. 케이스마다 *처리 방식*과 *라운드 2 미루기* 칸을 같이 두면, 라운드 2 자동화 룰을 만들 때 본 문서가 그대로 입력값이 된다.
5. 본 문서는 *케이스 카탈로그*에 가깝다. 발견되는 새 케이스는 본문에 추가되며, 처리 방식이 정책으로 굳어지면 `04-policy.md` 본문에 흡수되어 본 문서에서 빠진다.

## 묶음 1 — 응시 중단·끊김

응시자가 시작 후 *시간이 끝나기 전*에 발생하는 단절. 정책 2(*응시자 조작 없음*) · 정책 3(*자동 저장·재전송*) 원칙을 깨지 않는다.

| # | 케이스 | 1차 오픈 처리 |
|---|---|---|
| E1-1 | 네트워크 일시 끊김 (5초~3분) | 오프라인 표시만 상단 안내. 응시 계속. 답안은 IndexedDB 임시 저장. 복구 시 자동 재전송 (정책 3) |
| E1-2 | 네트워크 끝까지 복구 안 됨 | 시험 종료 시 자동 일괄 전송 시도 → 실패 시 *시스템 장애 재응시 티켓* 흐름으로 흡수 (정책 6) |
| E1-3 | 응시 중 새 창 닫힘 (window.close 또는 강제 종료) | 마이페이지 *응시 가능 시험* 카드는 *진행 중* 상태 유지. 응시 윈도우 안에서 *동일 시험 재진입* 시 마지막 문항부터 이어서 진행. 이어서 응시 미허용 시(GAP-08 결정 전 가정)는 시스템 장애 재응시 티켓 |
| E1-4 | 응시 중 마이크 권한 도중 회수 | 응시자에게 *마이크 끊김* 토스트 1회. 다음 문항부터 자동 진행은 계속되지만 녹음은 빈 답안. 시험 종료 후 응시자 신고 또는 운영자 로그 확인 → 시스템 장애 케이스로 흡수 |
| E1-5 | 응시 중 전화·알람 인터럽션 | 모바일 OS의 마이크 권한 회수와 같은 경로 (E1-4)로 흡수 |

**라운드 2 이후**: 끊김 자동 감지 → 일시정지 후 재개, 인터럽션 정책 분기, 부분 전송 후 단말 변경 시나리오.

## 묶음 2 — 환경·기기 변화

환경 점검은 통과했지만 *응시 시작 시점*에 기기 상태가 달라진 케이스. 정책 1(*사전 점검 통과 데이터가 없으면 결제 버튼 비활성*) 원칙을 응시 직전까지 끌고 갈지에 대한 답.

| # | 케이스 | 1차 오픈 처리 |
|---|---|---|
| E2-1 | 환경 점검 30일 캐시 적중 + 응시 시작 직전 마이크 망가짐 | 본인 확인 → 환경 점검 페이지 진입 시 *마이크 신호 5초 자동 확인*. 신호 없으면 응시 시작 버튼 비활성. 응시자가 마이크 교체 후 재테스트 |
| E2-2 | 환경 점검 통과 후 응시 시작했는데 1형부터 마이크 신호 0 | 1형 종료 시점에 자동 감지 → 응시자에게 *마이크 신호 없음* 안내 후 응시 종료. 시스템 장애 케이스로 운영자가 재응시 티켓 발행 (정책 6) |
| E2-3 | 응시 도중 디바이스 변경 시도 (포털 데스크탑 → 사용자 web 모바일) | 1차 오픈 미허용. *동일 시험 1세션 1디바이스* 원칙. 시도 시 새 디바이스 진입 거절 안내 |
| E2-4 | 동일 계정으로 두 디바이스 동시 응시 시도 | 뒤에 들어온 세션 거절. 먼저 들어온 세션 유지 |
| E2-5 | 응시 도중 디바이스 OS 강제 업데이트·재부팅 | E1-3(새 창 닫힘)과 같은 경로로 흡수 |

**라운드 2 이후**: 디바이스 핑거프린팅 기반 자동 재테스트, 마이크 변경 감지, 응시 시작 직전 환경 재테스트 강제 여부 정책 결정 (GAP-08).

## 묶음 3 — 결제·정산 정합성

응시자 입장에서 *돈은 빠져나갔는데 응시 못 하는 상태*. 가장 클레임이 거센 영역. 결제 모듈(PG·GAP-12 결정 후)이 자동 회수해야 하는 것과 운영이 수동 보정하는 것을 가른다.

| # | 케이스 | 1차 오픈 처리 |
|---|---|---|
| E3-1 | PG 결제 성공 → 콜백 실패 → 마이페이지에 시험 안 보임 | 결제 모듈이 PG 트랜잭션 정합성 주기 점검(예: 5분 간격)으로 자동 보정. 24시간 내 미보정 시 응시자가 1:1 문의 → 운영자 수동 등록 |
| E3-2 | 이중 결제 (응시자 새로고침으로 동일 시험 2회 결제) | 결제 모듈에서 자동 회수 (정책 7 — PG 트랜잭션 정합성). 이중 청구 잔류 시 환불 흐름(E3-3)으로 흡수 |
| E3-3 | 응시 전 환불 신청 (접수 기간 내) | 마이페이지 환불 요청 → 운영 검토 → 승인 시 PG 환불 호출. 응답 SLA *영업일 N일* (정책 7) |
| E3-4 | 응시 후 환불 신청 (단순 변심) | 미적용. 마이페이지에서 신청 자체를 차단 또는 *환불 불가* 안내 (정책 7) |
| E3-5 | 결제는 됐는데 응시 윈도우(14일) 안에 미응시 | 자동 만료. 환불 없음 (정책 8). 만료 임박 D-1·D-day 리마인더 메일 (`03-user-flows.md` §6) |

**라운드 2 이후**: 자동 환불 분기(접수 기간 내 + 미응시 → 자동 승인), 환불 사유 코드 매트릭스, 적격청구서 인보이스 PDF.

## 묶음 4 — 윈도우·시간 경계

*결제 후 14일* 윈도우(정책 8)와 응시 윈도우 안 *24시간 응시 가능 티켓*의 시간 경계.

| # | 케이스 | 1차 오픈 처리 |
|---|---|---|
| E4-1 | 윈도우 마지막 날 자정 직전 응시 시작 → 응시 도중 자정 도과 | 시작 시점 기준으로 응시권을 잠근다. 응시 도중 자정 도과 무관하게 끝까지 진행 |
| E4-2 | 윈도우 만료 직전 마이페이지 진입했는데 응시 시작 안 함 | 만료 시각에 자동 만료. 환불 없음. 만료 후 응시 시도는 *기간 만료* 안내 |
| E4-3 | 결제 직후 응시 시작 — 시스템 시각 vs 응시자 로컬 시각 불일치 | 시스템 시각(JST) 기준 단일 진실. 응시자 로컬 시각 무시 |
| E4-4 | 응시 중 SSO 토큰 만료 | 토큰 자동 갱신. 갱신 실패 시 응시는 계속(로컬 IndexedDB 저장), 시험 종료 후 재전송 시점에 재로그인 유도 |
| E4-5 | 응시 윈도우 안 *재진입* 시 일부 답안만 전송된 상태 | 마지막 전송된 문항 다음부터 이어서 진행. 이전 문항 재응시 미허용 (정책 2 *응시자 조작 없음* 원칙 일관성) |

**라운드 2 이후**: 응시 일시정지/이어서 응시 정책, 시간대(타임존) 이동 응시자 처리, 토큰 만료 자동 모달.

## 묶음 5 — 운영자 누락·SLA

운영이 *수동 처리해야 하는데 빼먹은 경우*. 1차 오픈에서 가장 명확한 SLA가 필요한 영역.

| # | 케이스 | 1차 오픈 처리 (운영 SLA) |
|---|---|---|
| E5-1 | AI 1차 채점 완료 → 운영자가 *결과 공개* 안 누름 | 채점 완료 후 *영업일 N일* 안에 공개 (N = 운영팀 캐파 산정). 미공개 시 관리자 대시보드 상단 카운터로 누락 건수 표시 |
| E5-2 | 환불 신청 → 운영자가 검토 안 함 | 신청 후 *영업일 N일* 안에 승인·반려 (정책 7). 미처리 시 관리자 대시보드 상단 카운터 |
| E5-3 | 1:1 문의 답변 누락 | 문의 접수 후 *영업일 N일* 안에 답변. 미답변 시 관리자 대시보드 카운터 |
| E5-4 | 일본 연휴·주말 운영 공백 | SLA는 *영업일* 기준. FAQ·자동 응답 메일에 *영업일 = 일본 평일* 명시 |
| E5-5 | 시스템 장애 재응시 티켓 발행 누락 | 응시자 1:1 문의로 인입 시 우선 처리. SLA *영업일 N일* (E5-3과 동일) |

> **N값**은 운영팀(KOEN)이 1차 오픈 캐파 산정 후 결정. 본 문서는 *형식만* 박고 숫자는 비워둔다.

**라운드 2 이후**: SLA 자동 알림(이메일·인앱), 결과 공개 자동화(채점 후 X분), 운영자 등급별 권한 분리.

## 묶음 6 — 어뷰징·계정 무결성

회원 자격(정책 9) 안에서도 우회 시도가 발생하는 케이스.

| # | 케이스 | 1차 오픈 처리 |
|---|---|---|
| E6-1 | 한 계정에서 무료 샘플 반복 응시 | *계정당 샘플 1회*. 2회차 시도 시 *이미 응시함* 안내 + 본 시험 결제 CTA |
| E6-2 | 계정 다수 생성 후 무료 샘플 반복 | 이메일 도메인 단위 차단은 라운드 2. 1차 오픈은 *동일 IP에서 X개 이상 가입* 시 운영자 대시보드 알림만 |
| E6-3 | 응시 중 대리 응시 의심 | 1차 오픈에서 자동 감지 미운영. 본인 확인 체크박스(이름·이메일·시험명) + 약관 동의로 시작 (정책 4) |
| E6-4 | 결과 공개 후 SNS·블로그에 문제 유출 | 1차 오픈에서 자동 감지 미운영. 약관에 *문제 유출 시 응시 자격 박탈* 조항만 명시 |
| E6-5 | 회원 탈퇴 신청 — 미응시 시험·미사용 결제권 잔류 | 응시·결제 이력이 있으면 탈퇴 미허용. 응시권 사용 또는 환불 처리 후 탈퇴 가능 안내 |

**라운드 2 이후**: 디바이스 핑거프린팅 기반 다계정 감지, 부정행위 방지(감독 카메라·녹화), 회원 탈퇴 시 데이터 보존 정책(일본 개인정보보호법 기준).
